Executive Summary

The Opportunity

Takealot sellers are operating blind. They see revenue but have no clear visibility into actual profitability. By the time they discover a product is losing money, they've already restocked it three times.

Problem Statement

Currently, Takealot marketplace sellers determine profitability through a manual, time-consuming process:

  1. Export CSV files from the Takealot Seller Portal containing sales, orders, and fees
  2. Build complex spreadsheets with manual formulas to calculate margins
  3. Manually input COGS (Cost of Goods Sold) for each product
  4. Reconcile fees across multiple transaction types
  5. Discover losses retrospectively - often weeks after restocking unprofitable products
⏱️ Time Cost: The average seller spends 2-3 hours per week on profitability analysis
💸 Financial Cost: Hidden losses average R850-R2,400 per month per seller on products they didn't know were unprofitable

Database Schema Design

-- Core Tables CREATE TABLE sellers ( id UUID PRIMARY KEY, email VARCHAR(255) UNIQUE NOT NULL, takealot_api_token TEXT ENCRYPTED, takealot_refresh_token TEXT ENCRYPTED, vat_registered BOOLEAN DEFAULT false, target_margin_pct DECIMAL(5,2) DEFAULT 25.00, subscription_tier VARCHAR(50) DEFAULT 'free', created_at TIMESTAMP DEFAULT NOW(), last_sync TIMESTAMP ); CREATE TABLE products ( id UUID PRIMARY KEY, seller_id UUID REFERENCES sellers(id), tsin VARCHAR(100) NOT NULL, name VARCHAR(500), category VARCHAR(100), weight_kg DECIMAL(8,3), dimensions_cm3 INTEGER, landed_cost_cents INTEGER, -- Store in cents to avoid float issues barcode_exists BOOLEAN DEFAULT true, current_price_cents INTEGER, competitor_price_cents INTEGER, -- For Buy Box analysis last_updated TIMESTAMP DEFAULT NOW(), UNIQUE(seller_id, tsin) ); CREATE TABLE orders ( id UUID PRIMARY KEY, seller_id UUID REFERENCES sellers(id), product_id UUID REFERENCES products(id), takealot_order_id VARCHAR(100) UNIQUE, order_date TIMESTAMP NOT NULL, quantity INTEGER NOT NULL, selling_price_incl_vat_cents INTEGER NOT NULL, shipped_from_dc VARCHAR(10), -- 'JHB' or 'CPT' shipped_to_customer_location VARCHAR(10), status VARCHAR(50), -- 'confirmed', 'shipped', 'delivered', 'returned' created_at TIMESTAMP DEFAULT NOW() ); CREATE TABLE fees ( id UUID PRIMARY KEY, order_id UUID REFERENCES orders(id), fee_type VARCHAR(50), -- 'success', 'fulfilment', 'storage', 'ibt', 'cancellation', 'subscription' amount_cents INTEGER NOT NULL, calculation_date TIMESTAMP DEFAULT NOW(), details JSONB -- Store breakdown for complex fees ); CREATE TABLE profitability_snapshots ( id UUID PRIMARY KEY, seller_id UUID REFERENCES sellers(id), product_id UUID REFERENCES products(id), period_start DATE NOT NULL, period_end DATE NOT NULL, units_sold INTEGER DEFAULT 0, total_revenue_cents INTEGER DEFAULT 0, total_fees_cents INTEGER DEFAULT 0, total_cogs_cents INTEGER DEFAULT 0, allocated_costs_cents INTEGER DEFAULT 0, -- Inbound shipping, ads, etc. net_profit_cents INTEGER DEFAULT 0, margin_pct DECIMAL(5,2) DEFAULT 0, created_at TIMESTAMP DEFAULT NOW(), UNIQUE(seller_id, product_id, period_start, period_end) ); CREATE TABLE alerts ( id UUID PRIMARY KEY, seller_id UUID REFERENCES sellers(id), product_id UUID REFERENCES products(id), alert_type VARCHAR(50), -- 'LOSS_MAKING', 'LOW_MARGIN', 'STORAGE_WARNING', 'IBT_PENALTY', etc. severity VARCHAR(20), -- 'info', 'warning', 'critical' message TEXT, recommendation TEXT, dismissed BOOLEAN DEFAULT false, created_at TIMESTAMP DEFAULT NOW() ); CREATE TABLE stock_levels ( id UUID PRIMARY KEY, seller_id UUID REFERENCES sellers(id), product_id UUID REFERENCES products(id), dc_location VARCHAR(10), -- 'JHB' or 'CPT' units_on_hand INTEGER DEFAULT 0, days_cover INTEGER DEFAULT 0, last_updated TIMESTAMP DEFAULT NOW(), UNIQUE(seller_id, product_id, dc_location) ); -- Indexes for performance CREATE INDEX idx_orders_seller_date ON orders(seller_id, order_date DESC); CREATE INDEX idx_products_seller ON products(seller_id); CREATE INDEX idx_alerts_seller_dismissed ON alerts(seller_id, dismissed); CREATE INDEX idx_profitability_seller_period ON profitability_snapshots(seller_id, period_start, period_end);

⚠️ API Limitations Discovered

Fees are often aggregated, not per-order. The /financial/transactions endpoint may return a daily summary like "Success Fees: R450" rather than "Order #12345: R15 success fee, Order #12346: R18 success fee."

Workaround: We'll calculate expected fees ourselves using the Takealot pricing schedule, then reconcile with actual fees daily to catch discrepancies.

Competitor Price Scraping - Technical Plan

Web Scraping Implementation

// Puppeteer scraper for Takealot product pages const puppeteer = require('puppeteer'); async function scrapeCompetitorPrices(tsin, productUrl) { const browser = await puppeteer.launch({ headless: true }); const page = await browser.newPage(); // Set realistic headers to avoid detection await page.setUserAgent('Mozilla/5.0 ...'); await page.goto(productUrl, { waitUntil: 'networkidle2' }); // Extract Buy Box winner const buyBoxPrice = await page.$eval( '.buying-options .price', el => parseFloat(el.textContent.replace(/[^0-9.]/g, '')) ); // Extract all seller prices (if "See all sellers" is clicked) await page.click('.all-sellers-link'); // If exists const allSellerPrices = await page.$$eval( '.seller-card .price', elements => elements.map(el => ({ price: parseFloat(el.textContent.replace(/[^0-9.]/g, '')), seller: el.closest('.seller-card').querySelector('.seller-name').textContent })) ); await browser.close(); return { buyBoxPrice, lowestCompetitorPrice: Math.min(...allSellerPrices.map(s => s.price)), averagePrice: allSellerPrices.reduce((a, b) => a + b.price, 0) / allSellerPrices.length, totalCompetitors: allSellerPrices.length, ourRanking: allSellerPrices.findIndex(s => s.seller === 'Our Seller') + 1 }; } // Run daily for all active products async function dailyCompetitorScan(sellerId) { const products = await db.getActiveProducts(sellerId); for (const product of products) { try { const competitorData = await scrapeCompetitorPrices( product.tsin, `https://www.takealot.com/product/${product.tsin}` ); await db.updateProduct(product.id, { competitor_lowest_price: competitorData.lowestCompetitorPrice, buy_box_price: competitorData.buyBoxPrice, competitor_count: competitorData.totalCompetitors, last_competitor_check: new Date() }); // Alert if undercut if (product.current_price > competitorData.buyBoxPrice + 5) { await createAlert({ type: 'UNDERCUT_BY_COMPETITOR', message: `You've been undercut on "${product.name}". Buy Box is now R${competitorData.buyBoxPrice} vs your R${product.current_price}`, recommendation: `Lower to R${competitorData.buyBoxPrice - 1} to win Buy Box` }); } } catch (error) { console.error(`Scraping failed for ${product.tsin}:`, error); // Fallback: Use API data only, mark as estimate } // Rate limiting: Wait 2-5 seconds between requests await sleep(Math.random() * 3000 + 2000); } }

Scraping Best Practices:

  • Rotate user agents to mimic real browsers
  • Add random delays between requests (2-5 seconds)
  • Use residential proxies if Takealot starts blocking (unlikely for occasional scraping)
  • Cache results for 24 hours (competitor prices don't change hourly)
  • Have fallback: if scraping fails, seller can manually input competitor price

Webhook Implementation

Takealot supports webhooks for real-time events. We'll subscribe to:

Webhook Event Trigger Our Action
order.created New order placed • Calculate profit instantly
• Update dashboard in real-time
• Send notification if margin < target
order.cancelled Order cancelled by customer or Takealot • Reverse profitability calculation
• If late cancellation, apply R50 penalty to P&L
• Alert seller to investigate
return.initiated Customer requests return • Adjust profit (assume refund will happen)
• Track return rate per product
• Alert if >10% return rate
stock.low Stock falls below threshold • Warn if approaching storage fee threshold (35 days)
• Suggest optimal restock quantity

Webhook Security

Takealot signs webhook payloads with HMAC-SHA256. We verify every incoming webhook:

const crypto = require('crypto'); function verifyWebhook(req, webhookSecret) { const signature = req.headers['x-takealot-signature']; const payload = JSON.stringify(req.body); const expectedSignature = crypto .createHmac('sha256', webhookSecret) .update(payload) .digest('hex'); if (signature !== expectedSignature) { throw new Error('Invalid webhook signature'); } return true; }

10.3 Critical Success Factors

What Must Go Right for This to Succeed

  1. The "Aha" Moment Must Happen Fast

    Within 60 seconds of connecting their Takealot account, sellers must discover something valuable (hidden loss, optimization opportunity). If onboarding takes >5 minutes or doesn't show clear value, we lose them.

  2. Accuracy is Non-Negotiable

    If our fee calculations are off by even 5%, sellers will lose trust immediately. We must match Takealot's actual fees exactly, with daily reconciliation.

  3. Real-Time Updates Create Addiction

    The dopamine hit of seeing "New sale: +R45 profit" in real-time creates a checking habit. Webhooks are critical, not optional.

  4. ROI Must Be Obvious and Quantified

    Don't say "helpful insights" - say "You avoided R850 in losses this month by not restocking these 3 products." Concrete numbers drive conversions.

  5. Speed to Market Wins

    First-mover advantage in this space is massive. Launch MVP in 6 weeks, iterate from real user feedback. Don't wait for perfection.
